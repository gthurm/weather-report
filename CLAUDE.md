# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build

```sh
make          # build the `sensors` binary
make clean    # remove binary and object files
```

Requires `libsqlite3-dev` (or equivalent) for `db.c`. All other dependencies are Linux kernel headers (`linux/i2c-dev.h`) and libc.

## Run

```sh
./sensors [period_seconds [db_path]]   # defaults: 5 s, sensors.db
```

Must run on the Raspberry Pi (or any Linux board with `/dev/i2c-1`). Loops indefinitely, sampling every `period_seconds`, updating the OLED, printing a JSON object to stdout, and inserting a row into the SQLite database. Exits cleanly on `SIGINT`/`SIGTERM` (clears the display).

## Hardware

| Device  | I2C addr | Provides                        |
|---------|----------|---------------------------------|
| BMP280  | `0x77`   | temperature (°C), pressure (Pa) |
| AHT20   | `0x38`   | temperature (°C), humidity (%)  |
| SSD1315 | `0x3C`   | 128×64 OLED display             |

All three share `/dev/i2c-1`. The main loop opens the bus once and each module calls `ioctl(fd, I2C_SLAVE, addr)` to switch targets before every transaction.

## Architecture

`sensors.c` is the only entry point. It orchestrates four independent modules:

- **`bmp280.c`** — opaque-handle driver. `bmp280_init` reads factory calibration registers once and caches them in the handle. `bmp280_sample` triggers a forced-mode measurement and applies the double-precision compensation formulas from the BMP280 datasheet (§4.2.3).
- **`aht20.c`** — single public function `aht20_sample(fd, &temp, &hum)`. Checks the calibration status bit on every call and re-initialises if needed before triggering a measurement.
- **`ssd1315.c`** — opaque-handle driver for the SSD1306/SSD1315 OLED. Maintains a full framebuffer in memory; `ssd1315_flush` streams all 1024 bytes in one I2C write. Bundles a 5×7 font covering ASCII 0x20–0x7E.
- **`db.c`** — SQLite persistence. `db_open` creates the `samples` table if absent and prepares the `INSERT` statement once. `db_insert` takes nullable `double *` pointers — `NULL` for any sensor that errored, which becomes a SQL `NULL` in the row. Timestamp is generated by SQLite (`strftime(...,'now')`).

Data flow per iteration: sample both sensors → update OLED → print JSON to stdout → insert DB row. Sensor errors are non-fatal throughout; they appear as `NULL` columns in the DB and as `"error": "read failed"` in JSON.
